# 기획 변경 검토 보고서

**날짜**: 2026-04-20
**검토 대상**: `spec.md`, `design-system.md` (commit `5bbc60e`)
**검토자**: Sisyphus (AI Orchestrator)

## 1. 변경 요약

프로젝트의 핵심 정체성이 **"Floating AI Prompt Assistant"**에서 **"Structured Prompt Builder"**로 전환되었습니다.

### Before → After

| 항목 | 이전 (v1) | 현재 (v2) |
|------|-----------|-----------|
| **제품명** | Floating AI Prompt Assistant | FloatingBoard Structured Prompt Builder |
| **핵심 가치** | 짧은 입력 → LLM 확장 → 자동 붙여넣기 | 구조화된 선택 → 프롬프트 조립 → 복사 |
| **시작점** | 자유 텍스트 입력 | 대주제 → 소주제 → 키워드 선택 |
| **LLM 역할** | 핵심 (프롬프트 생성 주체) | 보조 (다듬기/번역 선택 옵션) |
| **종착점** | Auto-Injection (타겟 앱에 Cmd+V) | 클립보드 복사 |
| **Skill 시스템** | 커스텀 프롬프트 템플릿 관리 | 고정 Taxonomy (대주제/소주제/키워드) |
| **스트리밍** | 필수 (SSE/NDJSON) | 불필요 (JSON POST 우선) |
| **Panel 크기** | 480pt (미니멀 입력창) | 560pt (빌더, 최대 620pt 높이) |
| **접근성 권한** | 필수 (CGEvent Auto-Injection) | 불필요 |

|-----------|------|------|

## 3. 효용성 분석

### 3.1 전환의 긍정적 효과

| 측면 | 효과 | 근거 |
|------|------|------|
| **MVP 범위 명확성** | ★★★★★ | "코딩 하나만"이라는 명확한 스코프. 우선순위 판단이 쉬움 |
| **사용자 문제 정의** | ★★★★★ | "어떻게 잘게 나눠 생각해야 할지 모른다"는 통찰은 실제 페인포인트. 자유 입력형 도구와의 차별화가 선명 |
| **LLM 의존도 감소** | ★★★★☆ | API Key 없이도 기본 가치 성립 → 진입 장벽 하락, 오프라인 사용 가능 |
| **프롬프트 품질 일관성** | ★★★★☆ | 고정 골격 + 타입별 슬롯 → LLM에 의존하지 않고도 일정 수준 품질 보장 |
| **확장성** | ★★★★☆ | Taxonomy JSON만 교체하면 새 도메인(기획, 리서치 등)으로 확장. 조립 엔진 재사용 |

### 3.2 우려 사항

| 우려 | 심각도 | 분석 |
|------|--------|------|
| **Panel 크기 증가** | 중간 | 560pt 폭 × 최대 620pt 높이는 Spotlight(≈480×120)보다 훨씬 큼. 작업 영역 침범 가능. 다만 빌더 성격의 앱이므로 Raycast 수준의 크기는 수용 가능 |
| **키워드 수에 따른 UI 복잡도** | 높음 | 5개 그룹 × 그룹당 4~8개 키워드 = 20~40개 칩. 화면이 붐빌 수 있음 → 소주제별 필터링, 접기/펼치기, 기본 추천 노출 전략이 필수 |
| **선택 피로** | 중간 | 대주제 → 소주제 → 키워드 → 초안 → 다듬기 → 번역 → 복사는 7단계. 클릭 수가 많으면 "그냥 직접 쓰는 게 낫다"는 반응 가능 → 스마트 기본값, 이전 선택 기억으로 마찰 최소화 필요 |
| **고정 골격의 경직성** | 중간 | 9슬롯 고정 구조가 모든 상황에 맞지 않을 수 있음. "기능 삭제" 작업에서 verification 슬롯이 불필요할 수도 있음 → 소주제별 슬롯 활성/비활성 분기 설계 필요 |
| **Prompt 다듬기의 실제 가치** | 낮음 | 조립 결과가 이미 충분히 좋으면 LLM 다듬기의 한계 효용이 낮을 수 있음. 반대로 조립 품질이 낮으면 LLM이 구워삶아도 한계. → 핵심 가치는 조립 엔진 품질에 있고, LLM은 체리옵 정도로 위치잡는 게 맞음. 현재 기획이 이미 이 방향 |
| **Auto-Injection 제거의 사용성 영향** | 낮음 | Cmd+V 한 번이면 되는 작업. Auto-Injection은 "마법 같은 느낌"은 있었지만, 엣지 케이스(권한, 앱 충돌)가 많았음. 복사 방식이 더 안정적 |

### 3.3 핵심 질문에 대한 평가

**Q: "이 변경이 제품의 가치를 높이는가?"**

**네.** 이전 기획은 "더 나은 프롬프트 입력기"였고 경쟁이 치열합니다 (Raycast AI, Cursor Cmd+K, ChatGPT). 현재 기획은 "사고 분해 도구"라는 새로운 카테고리를 만듭니다. 차별화가 명확하고, 해결하는 문제가 구체적입니다.

**Q: "MVP 출시 속도가 빨라지는가?"**

**네, 하지만 Phase 1은 느려집니다.** 전체 라인업의 구현 볼륨은 줄었지만(Auto-Injection, SSE 제거), Phase 1의 UI 복잡도는 증가했습니다. 칩 기반 빌더 UI는 단순 입력창보다 구현량이 많습니다. 다만 Phase 3(LLM)이 단순화되어 전체 타임라인은 단축됩니다.

**Q: "LLM 없이 가치가 성립하는가?"**

**네.** 이것이 이번 전환의 가장 큰 승점입니다. 고정 골격 조립만으로도 "직접 쓰는 것보다 나은 프롬프트"가 나옵니다. LLM은 부가 가치입니다.

## 6. 종합 평가

| 측면 | 평점 | 한 줄 평 |
|------|------|---------|
| **기획 방향** | ★★★★★ | 문제 정의가 정확하고 솔루션이 차별적. "사고 분해"라는 프레이밍이 강력 |
| **구현 난이도** | ★★★☆☆ | Phase 1 UI 복잡도 증가. 전체적으로는 Auto-Injection/SSE 제거로 하락 |

**총평**: 기획 전환은 **올바른 방향**입니다. "더 나은 프롬프트 입력기"에서 "사고 분해 도구"로의 이동은 경쟁 우위를 만듭니다. 구현 부담도 전체적으로 감소했습니다. Phase 1의 UI 복잡도 관리(Taxonomy 품질, 키워드 노출 전략)가 성패를 가릅니다.
